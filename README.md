# Subscriptions Service

REST-сервис для агрегации данных об онлайн подписках пользователей.

## Запуск

```bash
cp .env.example .env
docker compose up --build
```

Сервис будет доступен на `http://localhost:8080`.

## Swagger UI

```
http://localhost:8080/swagger/index.html
```

Перегенерировать Swagger после изменения хэндлеров:

```bash
swag init -g cmd/main.go -o docs
```

## API

| Метод  | Путь                             | Описание                     |
|--------|----------------------------------|------------------------------|
| POST   | /api/v1/subscriptions            | Создать подписку             |
| GET    | /api/v1/subscriptions            | Список подписок пользователя |
| GET    | /api/v1/subscriptions/{id}       | Получить подписку по ID      |
| PUT    | /api/v1/subscriptions/{id}       | Обновить подписку            |
| DELETE | /api/v1/subscriptions/{id}       | Удалить подписку             |
| GET    | /api/v1/subscriptions/total-cost | Суммарная стоимость подписок |

> **`total-cost`**: параметры `from` и `to` задают включительный период. Например, `from=01-2025&to=03-2025` охватывает январь, февраль и март 2025.

## Тесты

### Unit-тесты сервисного слоя

Покрывают всю бизнес-логику через ручной мок репозитория. Для запуска не нужна база данных.

```bash
go test ./internal/service/...
```

### Интеграционные тесты репозитория

Проверяют SQL-логику `TotalCost` против реального PostgreSQL. Контейнер поднимается автоматически через testcontainers-go — нужен только запущенный Docker.

```bash
go test ./internal/repository/... -timeout 120s
```

### Все тесты

```bash
go test ./... -timeout 120s
```

## Архитектурные решения

### Слои приложения

```
cmd/            — точка входа, инициализация зависимостей
internal/
  config/       — загрузка конфигурации из переменных окружения (cleanenv)
  domain/       — доменные модели и DTO
  transport/    — HTTP-хэндлеры, роутинг, middleware
  service/      — бизнес-логика
  repository/   — работа с базой данных
  apperr/       — типизированные ошибки приложения
pkg/
  postgres/     — настройка пула соединений pgxpool
migrations/     — SQL-миграции, встроенные в бинарник через embed.FS
```

Зависимости направлены строго внутрь: `transport → service → repository`. Каждый слой общается с соседним через интерфейс, что позволяет менять реализации независимо.

### Ключевые решения

**Интерфейсы между слоями.** `transport` знает только об интерфейсе `SubscriptionService`, `service` — только об интерфейсе `SubscriptionRepository`. Конкретные реализации подставляются в `main.go` через dependency injection.

**Типизированные ошибки.** Пакет `apperr` определяет коды ошибок (`INVALID_INPUT`, `NOT_FOUND`, `INTERNAL_ERROR`), которые сервис возвращает хэндлеру. Хэндлер маппит коды в HTTP-статусы. Это исключает магические числа и позволяет единообразно обрабатывать ошибки.

**Дата как месяц и год.** Согласно ТЗ, дата подписки хранит только месяц и год. В БД используется тип `DATE`, значение всегда приводится к первому числу месяца при сохранении. Формат в API — `MM-YYYY`.

**Подсчёт стоимости на стороне БД.** Запрос `total-cost` вычисляет суммарную стоимость одним SQL-запросом с помощью `SUM` и арифметики над датами, без загрузки записей в память.

**Миграции встроены в бинарник.** SQL-файлы подключаются через `embed.FS` и применяются при старте через `golang-migrate`. Отдельного шага деплоя для миграций не требуется.

**Конфигурация через переменные окружения.** `cleanenv` читает env-переменные с дефолтными значениями. В docker-compose переменные подставляются из `.env`-файла.
